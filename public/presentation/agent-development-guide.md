# 内网大代码库 Agent 开发建议

> 适用场景：内网模型（如 GLM 4.7 / Qwen / DeepSeek 等）、上下文有限、工具能力有限、Java 后端大项目、Oracle / MyBatis / Spring / WebLogic 等企业系统。
>
> 核心目标：尽量减少上下文消耗，让 Qwen Code CLI / OpenCode 这类 agent 稳定地开发新功能或修改老功能。

---

## 1. 总原则

不要让 agent 一上来就“理解整个系统”。

正确做法是：

```text
先缩小范围
再找证据
再写方案
再小步修改
最后复查 diff
```

核心原则：

1. 不允许一开始读取整个 `src` 目录。
2. 不允许直接实现需求，必须先判断是“新功能”还是“老功能修改”。
3. 新功能优先找同类模板，最多读取 1-2 个最相似实现。
4. 老功能必须先做只读影响分析，不允许直接修改。
5. 每次只处理一个模块。
6. 每次只修改一个层次：Controller / Service / Mapper / Test。
7. 每个结论必须带文件路径、类名、方法名或 Mapper ID。
8. 搜索结果太多时，先缩小关键词，不要把全部结果塞进上下文。
9. 阶段结束必须写报告文件。
10. 新阶段优先开新会话，只读取报告文件。
11. 修改前必须列出“要改文件”和“不改文件”。
12. 禁止顺手重构、格式化无关文件。

---

## 2. 新功能 vs 老功能修改

### 2.1 新功能通常更简单

新功能的上下文成本较低，因为可以尽量隔离：

```text
新 Controller
新 Service
新 DTO
新 Mapper
新表 / 新字段
```

新功能最适合使用：

```text
模板驱动开发
```

也就是先在项目里找一个最像的已有功能，然后模仿它的结构。

### 2.2 老功能修改更危险

老功能修改的难点不在“改哪几行”，而在：

```text
入口在哪里？
谁调用了它？
是否影响审批流？
是否影响历史查询？
是否影响批处理？
是否影响缓存？
是否影响 Mapper resultMap？
是否影响 Oracle SQL 性能？
```

老功能最适合使用：

```text
侦察驱动开发
```

也就是先只读分析影响范围，再小步修改。

## 3. 推荐工作模式

### 3.1 三阶段会话模型

不要把探索、规划、修改混在一个会话里。

推荐分成三类会话：

```text
1. 探索会话：只读代码，找证据，不改代码
2. 规划会话：只读探索报告，生成方案
3. 执行会话：只读方案和少量文件，小步修改
```

阶段之间尽量使用报告文件交接：

```text
.agent/reports/impact.md
.agent/reports/design.md
.agent/reports/review.md
```

这样可以减少历史聊天记录占用上下文。

## 4. 新功能开发协议

### 4.1 流程

```text
阶段 1：找模板
  - 只读
  - 找 1-2 个最像的已有功能

阶段 2：做方案
  - 基于模板生成文件清单
  - 明确新增/修改范围

阶段 3：分层实现
  - DTO / Request / Response
  - Controller
  - Service
  - Mapper / XML
  - Test

阶段 4：检查
  - 编译
  - 单测
  - Diff review
```

### 4.2 新功能第一轮提示词

```text
你现在是“新功能开发 agent”。

任务：新增【xxx功能】。

当前只做阶段 1：寻找模板。

工作方式：
1. 先寻找 1-2 个最相似的已有功能作为模板。
2. 不要扫描整个项目，只搜索和关键词相关的 Controller、Service、Mapper、XML。
3. 找到模板后，先输出实现计划，不要立刻改代码。
4. 计划必须包含：新增文件、修改文件、复用的已有模式、风险点、需要我确认的问题。
5. 未确认前不要修改代码。
6. 禁止顺手重构无关代码。
```

### 4.3 OpenCode 用法示例

```text
@explore 先不要改代码。请寻找和【xxx功能】最相似的已有实现。

只输出：
1. 候选 Controller
2. 候选 Service
3. 候选 Mapper
4. 请求参数对象
5. 返回对象
6. 接口路径，如果有
7. 最推荐模仿哪一个实现

不要读取无关模块。
```

### 4.4 新功能实现提示词

```text
基于下面这个已有功能作为模板，实现新的【xxx功能】。

约束：
1. 尽量复用项目现有风格。
2. 不要改动模板功能。
3. 新增文件/修改文件必须列清单。
4. 一次只完成一个层次：Controller / Service / Mapper / Test。
5. 不要顺手重构。
6. 如果发现需要修改其他层，先停止并说明原因。
```

## 5. 老功能修改协议

### 5.1 流程

```text
阶段 1：影响分析
  - 只读
  - 找入口、调用链、表、Mapper

阶段 2：修改方案
  - 只基于影响分析报告
  - 明确改 / 不改边界

阶段 3：小步修改
  - 一次一个文件或一层

阶段 4：回归检查
  - 编译
  - 单测
  - SQL 风险
  - Diff review
```

### 5.2 老功能影响分析提示词

```text
你现在是“老功能影响分析 agent”。

任务：分析【xxx老功能修改】会影响哪些代码。

严格规则：
1. 只读代码，不允许修改。
2. 不要生成最终方案。
3. 不要泛泛而谈。
4. 只输出有证据的内容。
5. 每个结论必须给出文件路径、类名/方法名/Mapper ID/表名。
6. 如果不确定，写“未确认”，不要猜。

请输出：
1. 入口列表
2. 调用链
3. 相关 Controller / Service / Mapper / XML
4. 相关表和字段
5. 可能影响的历史逻辑
6. 风险点
7. 下一步建议搜索关键词
```

### 5.3 OpenCode 用法示例

```text
@explore 只读分析【xxx老功能修改】影响范围。

要求：
1. 不要修改代码。
2. 找入口。
3. 找调用链。
4. 找相关 Mapper / XML / 表。
5. 输出文件路径和方法名。
6. 最后给出建议修改范围。
7. 没有证据的地方写“待确认”。
```

### 5.4 修改方案提示词

```text
请只基于下面的 impact.md 生成修改方案。
不要假设未出现的文件。
如果证据不足，标记为“待确认”。

输出：
1. 修改目标
2. 修改文件清单
3. 每个文件为什么要改
4. 修改顺序
5. 数据库影响
6. 事务影响
7. Oracle / MyBatis 风险
8. 测试建议
9. 不做哪些修改
```

### 5.5 小步修改提示词

```text
只修改【文件名】。

目标：实现 design.md 中第【x.x】节。

严格规则：
1. 禁止修改其他文件。
2. 禁止顺手重构。
3. 禁止格式化无关代码。
4. 如果发现需要改其他文件，先停止并说明原因。
5. 修改后输出变更摘要和风险点。
```

## 6. 上下文控制策略

### 6.1 不要塞这些内容

```text
整个 src 目录
整个模块目录
所有历史聊天记录
所有搜索结果
所有公司规范
所有 skill 文档全文
无关日志
无关测试文件
```

### 6.2 应该塞这些内容

新功能：

```text
需求
1-2 个相似模板功能
项目编码约定
相关表结构
少量公共工具类
```

老功能修改：

```text
需求
入口文件
调用链文件
Mapper XML
表结构
影响分析报告
当前 diff
```

### 6.3 上下文使用阈值

建议：

```text
低于 50%：正常工作
50% - 60%：准备阶段总结
60% - 80%：停止扩展搜索，写报告或压缩
超过 80%：不要继续开发，先 /compress、/clear 或开新会话
超过 90%：立即停止，让 agent 写交接报告
```

## 7. Qwen Code CLI 使用建议

### 7.1 常用命令

```text
/context          查看上下文使用情况
/context detail   查看更详细的上下文占用
/compress         压缩历史上下文
/clear            清理当前会话
/init             初始化项目上下文
/summary          生成项目总结
```

### 7.2 Qwen Code 阶段切换建议

探索结束后，不要直接继续实现。

推荐：

```text
1. 让 agent 写 impact.md
2. /compress 或 /clear
3. 新阶段只读 impact.md
4. 再生成 design.md
5. 再小步修改
```

### 7.3 压缩前提示词

```text
请先总结当前会话，保留以下内容：
1. 原始需求
2. 已确认相关文件
3. 调用链
4. 表和字段
5. 修改计划
6. 风险点
7. 待确认项
8. 下一步建议

不要保留无关搜索过程。
```

然后再执行：

```text
/compress
```

### 7.4 不要乱用 @ 目录

不要这样：

```text
@src
@src/main/java
```

更推荐：

```text
@src/main/java/.../ParamService.java
@src/main/resources/mapper/ParamMapper.xml
```

或者先用命令搜索：

```text
!rg -n "ParamService|T_PARAM_MAIN|effectiveDate|approve" src/main
```

再只注入关键文件。

## 8. OpenCode 使用建议

### 8.1 推荐职责

OpenCode 更适合做：

```text
找入口
找调用链
找同类模板
做影响分析
多 session 探索
```

尤其适合使用 `@explore` 做只读侦察。

### 8.2 推荐流程

```text
@explore 只读分析
  ↓
保存影响报告 impact.md
  ↓
Plan agent 生成修改计划
  ↓
Build agent 单步实现
  ↓
再用 @explore 或 reviewer 复查 diff
```

### 8.3 Explore 提示词模板

```text
@explore 只读分析这个需求：
【需求内容】

要求：
1. 不要改代码。
2. 找入口。
3. 找调用链。
4. 找相关 Mapper / XML / 表。
5. 每个结论必须给文件路径和方法名。
6. 如果没有证据，写“待确认”。
7. 最后输出建议修改范围和下一步搜索关键词。
```

## 9. 推荐组合打法

### 9.1 OpenCode 负责“找路”

```text
@explore 找入口
@explore 找调用链
@explore 找同类模板
@explore 做影响分析
```

### 9.2 Qwen Code CLI 负责“干活”

```text
按设计改一个文件
补一个 Mapper
写一个 DTO
生成一个测试
做一次代码审查
```

当然，如果某个工具接内网模型更稳定，就优先使用更稳定的那个。

---

## 10. 尽量把老功能修改转化成“新增旁路”

对企业系统，尤其是银行系统，能新增就不要大改核心旧逻辑。

优先级：

```text
1. 新增配置项控制新逻辑
2. 新增策略类，老逻辑不动
3. 新增方法，旧方法保留
4. 小范围改旧方法
5. 大范围重构旧流程
```

这样更容易：

```text
回滚
灰度
解释
测试
审计
```

## 12. Oracle / MyBatis 专项检查清单

### 12.1 Oracle SQL

涉及 SQL 时必须检查：

```text
1. 是否存在隐式类型转换
2. CHAR(8) 日期字段是否和 DATE 直接比较
3. 是否影响索引使用
4. 是否涉及分区键
5. 是否可能导致全表扫描
6. UPDATE 是否正确处理 NULL
7. 是否涉及 CLOB / LOB 表空间
8. 是否需要新增索引
9. 是否需要刷新统计信息
10. 是否影响历史数据
```

### 12.2 MyBatis

涉及 Mapper XML 时必须检查：

```text
1. if test 是否存在类型问题
2. 字符串和数字比较是否正确
3. resultMap 是否漏字段
4. insert / update 是否同步新增字段
5. 动态 SQL 是否可能多逗号或少条件
6. mapper 方法参数名是否和 XML 一致
7. collection / association 是否影响性能
8. 是否存在 N+1 查询风险
```

## 13. Diff Review 提示词

修改完成后，不要只让 agent 说“完成了”。

应该让它复查 diff：

```text
请审查当前 diff。

要求：
1. 按文件列出修改内容。
2. 判断是否超出 design.md 的范围。
3. 检查是否有无关格式化。
4. 检查是否遗漏 Mapper / resultMap / DTO / SQL。
5. 检查是否存在 Oracle / MyBatis 风险。
6. 给出建议测试用例。
7. 如果发现风险，不要继续修改，先输出 review.md。
```

## 14. 常见反模式

### 14.1 直接说“帮我实现”

错误：

```text
帮我实现参数新增字段功能。
```

更好：

```text
先只读分析参数新增字段会影响哪些 Controller、Service、Mapper、表字段。
不要修改代码。
每个结论给文件路径和方法名。
```

### 14.2 一次性让 agent 改完所有层

错误：

```text
按方案把 Controller、Service、Mapper、测试都改完。
```

更好：

```text
只修改 DTO 和 resultMap。
禁止修改 Service。
修改后输出 diff 摘要。
```

### 14.3 把搜索结果全塞进去

错误：

```text
这是 rg 的 2000 行结果，请分析。
```

更好：

```text
请根据前 50 个最相关命中判断候选文件。
先输出需要深读的 10 个文件，不要直接分析全部结果。
```

### 14.4 让 agent 自己决定全部范围

错误：

```text
你看着改吧。
```

更好：

```text
只能修改以下文件：
1. A.java
2. BMapper.xml
如果需要新增文件或修改其他文件，先停止并说明原因。
```

## 15. 一个完整示例

需求：

```text
参数设置新增“是否允许跨机构使用”，审批通过后生效，历史查询要展示。
```

### 第 1 轮：OpenCode Explore

```text
@explore 只读分析：
参数设置新增“是否允许跨机构使用”，审批通过后生效，历史查询要展示。

请找：
1. 参数新增/修改入口
2. 审批通过回调入口
3. 参数生效逻辑
4. 历史查询逻辑
5. 相关 Mapper XML
6. 相关表和字段

不要修改代码。
每个结论必须给文件路径和方法名。
```

### 第 2 轮：整理 impact.md

```text
请把刚才的分析整理成 .agent/reports/impact.md：
1. 确认相关文件
2. 可能修改点
3. 风险点
4. 待确认项
5. 建议修改顺序
```

### 第 3 轮：Qwen Code 生成 design.md

```text
@.agent/reports/impact.md

基于 impact.md 生成修改方案。
要求：
1. 优先新增字段和新增分支，不重构旧逻辑。
2. 一次只改一个层次。
3. 明确每个文件的修改原因。
4. 给出测试清单。
5. 明确不修改哪些范围。
```

### 第 4 轮：执行第一步

```text
只修改参数 DTO / entity / resultMap 相关字段。
禁止修改 Service 逻辑。
修改后输出变更文件清单和 diff 摘要。
```

### 第 5 轮：执行第二步

```text
只修改 Mapper insert / update / select，使新字段可保存和查询。
禁止修改审批逻辑。
```

### 第 6 轮：执行第三步

```text
只修改审批通过后的参数生效逻辑。
如果需要改其他文件，先停止说明。
```

### 第 7 轮：复查 diff

```text
请审查当前 diff，确认：
1. 是否超出 design.md 范围。
2. 是否遗漏 Mapper / resultMap / DTO。
3. 是否影响历史查询。
4. 是否存在 Oracle / MyBatis 风险。
5. 需要补哪些测试。
```

## 16. 最后总结

推荐心法：

```text
新功能：模板驱动
老功能：侦察驱动
上下文：报告接力
修改代码：小步补丁
风险控制：Diff Review
```

一句话：

```text
不要让 agent 一次性当架构师、开发、测试、评审。
让它一次只干一种活，并且每一步都留下证据。
```
