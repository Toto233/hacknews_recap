# Loop Engineering：从提示词到循环设计

> 原文：Addy Osmani, [Loop Engineering](https://addyosmani.com/blog/loop-engineering/)  
> 这页是 vibe coding 演讲补充资料的中文导读与要点整理，帮助现场理解原文观点。建议结合原文阅读。

## 一句话概括

“Loop Engineering”讨论的是：不要只把 AI coding agent 当成一个需要你一轮轮提示的工具，而是把注意力转向**设计一套会自己发现任务、分配任务、验证结果、记录状态并继续下一步的工作循环**。

也就是说，人的角色从“每一步都亲自 prompt agent”转为“设计、约束、观察和校准这个 prompt agent 的系统”。

## 为什么这和 vibe coding 有关

传统的 AI 编程协作更像这样：

1. 人提出一个需求；
2. agent 给出代码或建议；
3. 人阅读结果，再补充下一条指令；
4. 如此反复，直到可用。

Loop Engineering 想强调的是：当 agent 能调用工具、创建分支、运行测试、读取 issue、写 PR、调用子 agent 互相评审时，工作流会变成一个“小型生产系统”。人不再只是写提示词，而是在设计这个系统的边界、节奏、记忆和停止条件。

这和 vibe coding 的关系在于：vibe coding 不是“把脑子交给模型”，而是把粗糙意图快速外化，再用一套可验证的循环把结果拉回工程现实。

## 文章里的核心结构：五个组件 + 一个状态层

原文把一个有效循环拆成五个主要组件，再加一个外部状态层。

### 1. Automations：让循环真的循环起来

自动化负责按计划触发任务，例如每天扫描：

- 昨天 CI 为什么失败；
- 最近有哪些 issue 需要 triage；
- 有没有新提交引入了风险；
- 哪些重复性修复可以交给 agent 先起草。

关键点不是“定时跑一个 prompt”这么简单，而是让系统在固定节奏下发现工作、记录结果，并把需要人判断的部分送回给人。

### 2. Worktrees：隔离并行工作

一旦同时跑多个 agent，最大的问题不是它们不够聪明，而是它们会改到同一批文件，互相踩踏。

Git worktree 的意义是：每个 agent 在独立工作目录和分支里修改代码。这样可以让多个任务并行进行，同时降低文件冲突和上下文污染。

但隔离只能解决机械冲突，不能解决人的评审瓶颈。你能同时启动多少 agent，不取决于工具按钮有多少，而取决于你能认真 review 多少结果。

### 3. Skills：把项目知识写成可复用说明

Skills 的作用是把项目约定、构建步骤、历史坑点和团队偏好写下来，避免每次对话都让模型从零猜。

一个好的 skill 会告诉 agent：

- 这个项目怎么构建和测试；
- 哪些目录不能碰；
- 常见需求应该走哪条路径；
- 过去踩过哪些坑；
- 什么样的结果才算完成。

这能减少“意图债”：如果你没有把约束写出来，agent 就会用自信但未必正确的方式填补空白。

### 4. Plugins / Connectors：连接真实工具链

只会读写本地文件的 agent 很有限。连接器让 agent 能接入真实工程环境，例如：

- issue tracker；
- 数据库或 staging API；
- Slack / 飞书 / 邮件；
- GitHub PR 和 CI；
- 内部文档和知识库。

这会把 agent 从“给出建议”推进到“能在真实流程中完成动作”：开 PR、关联工单、等待 CI、把结果通知到团队频道。

### 5. Sub-agents：把“写的人”和“查的人”分开

原文特别强调，写代码的 agent 不应该独自判断自己是否完成。

更稳的结构是：

- 一个 agent 负责探索或实现；
- 一个 agent 负责审查；
- 必要时再让第三个 agent 对照规格或测试做验证。

这并不是为了制造复杂度，而是为了避免同一个模型对自己刚生成的结果过于宽容。尤其当循环无人值守运行时，一个可信的 verifier 是你敢让它继续跑下去的前提。

### 6. State：循环必须有外部记忆

长时间运行的 agent 不能只依赖聊天上下文。它需要一个外部状态层，例如：

- Markdown 进度文件；
- Linear / GitHub issue；
- 项目看板；
- 任务日志；
- AGENTS.md / README / skill 文件。

状态层记录：做过什么、试过什么、失败在哪里、下一步是什么。没有这个层，循环每次都会像失忆一样重新开始。

## 一个典型循环长什么样

可以把它想成这样一条链路：

1. 每天早上自动化任务启动；
2. 它调用 triage skill，读取 CI、issue、近期 commit；
3. 它把发现写入 Markdown 或看板；
4. 对值得处理的问题，为每个任务开独立 worktree；
5. 一个子 agent 起草修复；
6. 另一个子 agent 对照项目 skill、测试和规范审查；
7. 连接器负责开 PR、更新工单、等待 CI；
8. 无法自动处理的部分进入 triage inbox，交给人判断；
9. 状态文件保留所有尝试，第二天继续接上。

这时你做的不是“连续写十条 prompt”，而是设计一套能反复运行的工程循环。

## 但循环不会替你承担工程责任

原文最重要的提醒是：循环会改变工作方式，但不会取消工程师的责任。

### 验证仍然是人的责任

“agent 说完成了”不是证明。你仍然要确认：

- 测试真的通过；
- 代码真的符合需求；
- 变更范围没有失控；
- 线上风险可接受；
- 你理解它为什么这么改。

### 理解会腐烂

循环越顺滑，越容易快速合入你没有读懂的代码。久而久之，代码库里会积累大量“存在但无人理解”的逻辑，这就是理解债。

### 舒服的姿势最危险

如果你设计循环只是为了逃避思考，它会放大问题；如果你设计循环是为了把重复劳动自动化，并把人类判断放在关键节点，它才会放大能力。

同样是“让循环跑起来”，结果可能完全相反。

## 对 vibe coding 演讲的补充理解

这篇文章可以补充三个观点：

1. **vibe coding 不是随便让模型写代码**：真正可用的 vibe coding 必须接上验证、隔离、状态和审查。
2. **提示词能力会变成系统设计能力**：未来的杠杆点不只是“怎么写一个好 prompt”，而是“怎么让一组 prompt、工具和 agent 形成可靠循环”。
3. **人不能从工程链路里消失**：你可以把很多步骤交给 agent，但需求判断、风险判断和最终验收仍然属于工程师。

## 可以带走的实践清单

如果要在自己的项目里试一版 Loop Engineering，可以从小开始：

- 先写一个 `SKILL.md` 或项目 agent 指南，记录构建、测试和禁区；
- 把一个重复任务做成每日/每周自动化检查；
- 给自动化任务一个明确的停止条件，例如“相关测试通过且 lint 通过”；
- 让实现 agent 和 review agent 分开；
- 所有自动化尝试都写入一个外部状态文件；
- 人只批准自己读过、跑过、理解过的改动。

## 结论

Loop Engineering 的重点不是让工程师消失，而是把工程师从逐条提示词的操作者，提升为循环系统的设计者。

真正的问题不再只是“我该怎么 prompt 这个 agent”，而是：

> 我该怎样设计一个循环，让 agent 能持续推进工作，同时仍然保留清晰的边界、验证和人类判断？

这也是它比 prompt engineering 更难、但杠杆更高的地方。
