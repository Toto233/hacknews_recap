#### 💡 Claude Code 实战 Tips

**CLAUDE.MD 文件的重要性：**

`CLAUDE.MD` 是 Claude Code 的"大脑"，写得越精确，效果越好。

**关键原则：**

1. **明确技术栈和约定**
   ```
   ❌ 错误：使用测试框架
   ✅ 正确：使用 pytest，测试文件命名规范为 test_*.py
   ```

2. **指定工具和命令**
   ```
   ❌ 错误：运行测试
   ✅ 正确：运行 pytest -v --cov=src
   ```

3. **代码风格约定**
   ```
   ❌ 错误：遵循 PEP 8
   ✅ 正确：使用 black 格式化，line length = 88，import 排序使用 isort
   ```

4. **项目结构说明**
   ```
   项目结构：
   - src/ 源代码
   - tests/ 测试代码
   - docs/ 文档
   - scripts/ 脚本工具
   ```

**版本控制安全原则：**

1. **禁止自动 commit/push**
   ```
   ❌ 错误：添加代码后自动提交
   ✅ 正确：只提供代码，让我审查后再决定是否提交
   ```

2. **明确审查要求**
   ```
   在 CLAUDE.MD 中声明：
   - 任何 commit/push 操作前必须获得明确授权
   - 先显示 diff，等待我确认
   - 重要修改使用 --no-verify
   ```

3. **分支策略**
   ```
   ❌ 错误：直接在 main 分支修改
   ✅ 正确：创建 feature/* 分支，完成后再合并
   ```

**实际 CLAUDE.MD 示例：**

```markdown
# 项目说明

## 技术栈
- Python 3.11
- FastAPI 0.104+
- PostgreSQL 15

## 开发约定

### 测试
- 使用 pytest
- 测试覆盖率目标：80%+
- 测试命令：pytest -v --cov=src

### 代码风格
- Black formatter (line length = 88)
- isort for imports
- 类型检查：mypy src/

### 版本控制
- ⚠️ 禁止自动 commit/push
- ⚠️ 任何 git 操作前必须获得明确授权
- ⚠️ 修改前先创建分支
- ⚠️ commit 前显示完整 diff 等待确认

### 项目结构
- src/ 核心代码
- tests/ 测试代码
- scripts/ 工具脚本
- docs/ 文档
```

**核心要点：**

- **精确性 > 通用性** - 具体约定越明确，AI 执行越准确
- **安全第一** - 明确禁止自动执行危险操作
- **人类在环** - 保留最终决策权
- **迭代优化** - 随着使用不断调整 CLAUDE.MD

### 4. 压轴：OpenClaw

#### 从 IDE 到个人助理

Claude Code 帮你写代码，OpenClaw 帮你做一切。

#### 模型切换：GLM-4.7 → ChatGPT 5.2（性格大变）

同一套使用方式，换模型之后“助理的性格”会明显变化：

- **GLM-4.7：** 更像一个美国人（更直接、更快、更偏执行）
- **ChatGPT 5.2：** 更像一个德国人（更严谨、更爱确认、更重视边界）

这几天折腾 OpenClaw 的过程本质上就是：**把这些差异固化成规则、写进文档、写进 memory**，让它越来越像“你自己的助理”。

#### 这几天我们具体折腾了什么（可讲的部分）

- **演讲站点的 slides 混乱**：同一个内容有好几套入口，改了 MD 但 slide 还是旧的。
  - 解决：只保留 `presentation/` 目录下 **2 个 MD**（讲稿 + slides 源），其余全部删掉，避免分叉。
- **线上页面 404/缓存**：某些链接看起来“老”，其实是路由没对应到静态文件，或者 CDN 还没刷新。
  - 解决：把演讲用的 Reveal 页面和 slides 源明确发布为静态资源。
- **QQ 邮箱能力波动**：改密码后授权失效，IMAP 登录失败。
  - 解决：重新生成授权码后再更新配置（这个你晚上回去搞）。
- **Win10 离线装 Python 包**：只下主包会触发 pip 联网拉依赖；飞书下载还会把 wheel 文件名改坏导致 pip 报错。
  - 解决：一次性打包依赖 wheels；并把 wheel 文件名改回规范格式（否则会提示 not a supported wheel）。

这些都属于 vibe coding 的“日常”：**不是写出一个完美系统，而是不断把坑填平、把规则固化**。

#### 真实案例：从"搞坏 WSL"到安卓手机上的 Dr.Two

我从今年2月初开始用 OpenClaw，到现在刚好一个月。

但这一个月里，我已经死了"一次"。

最开始部署在家里电脑的 WSL Ubuntu 上。

配置倒是挺快，但问题也很明显：

- 一开始什么都做不了，不知道让它干嘛
- 让它做点任务，结果把配置文件搞乱了
- 一来二去，直接把整个 WSL 搞坏了

再也起不来了。

##### 第二世：安卓手机上的 Dr.Two

后来，我换了个地方——安卓手机里的 Termux。

因为这次搞坏 WSL 的教训，这次我小心很多：

1. 配置文件改之前先备份
2. 重要的修改先在测试环境试
3. 建立了守护进程监控 gateway 状态

现在这个就是 Dr.Two——因为我已经重装过一次了。

##### 慢慢摸索的过程

第一个月，我慢慢摸索出它能做什么：

- 飞书收发消息、帮我写文档

- 定时检查邮件、天气提醒

##### 发现真相：强大的上限，羸弱的下限

在慢慢熟悉的过程中，我发现：

**OpenClaw 的上限极高。**

一旦你开始用，就会不断发现新的可能性：

- 今天只是发邮件
- 明天可以自动监控服务
- 后天可以帮你管理整个工作流

它像一个无限可能的平台，你想做什么，只要能描述清楚，它就能帮你做。

**但 OpenClaw 的下限也很羸弱。**

刚开始时：

- 配置文件搞错了，它直接崩溃
- 一句话没说清楚，它就做错了事
- 稍微复杂点的任务，它就不知道怎么处理

就像我第一个 WSL 那个版本，直接被我"用坏"了。

但关键在于：**这是一个会成长的东西。**

你用得越久，它越懂你。

配置越完善，它越稳定。

你让它做的越多，它越能处理复杂任务。

##### 我的建议：不要等"需求"

不要等到"我有明确需求了"再开始。

因为我一开始也不知道自己需要什么。

从"搞坏 WSL"到"离不开 Dr.Two"，中间没有任何界限。

唯一的区别是：**你是否开始用。**

##### 关键习惯：及时更新，固化知识

用 OpenClaw 的过程中，你会不断发现新工具、新用法。

**把它们写下来，固化到 MD 文件里。**

- 发现了好用的工具 → 写进 `TOOLS.md`
- 学到了新技巧 → 写进 `AGENTS.md` 或相关文档
- 遇到问题 → 写进 `MEMORY.md`

**为什么？**

- AI 每次会话都是新的，文件是它的"记忆"
- 你写下来，下次它就记得
- 积累多了，就是你自己的"AI 知识库"

这就是 vibe coding 的本质：**工具在帮你，你也在喂工具。**

### 5. 真正的千人千面：让你的大模型给你定制建议

传统时代，我们被归类到某个"类型"——内向型、外向型、理性型、感性型... 然后得到一包通用建议。

但 AI 时代不同了。

#### 一个 Prompt，无限可能

这是我问大模型的一个问题：

> **"2026 年，如果只选一个习惯——只要做到每天稳定执行，就会最大程度上改变你的人生——这个习惯是什么？"**

得到的建议是：**每天产出一个「可积累的确定性成果」**，每天留下一个"明天还能用"的东西。

可以是：
- 一段 100 行以内的代码
- 一篇 300–500 字的笔记
- 一个流程优化脚本
- 一条可复用的 Prompt
- 一页讲清楚一件事的文档

#### 关键不是建议本身

而是：**这是根据我的情况、我的背景、我的经历，专门生成的。**

你的回答会完全不同，因为：

- 你是前端工程师？大模型可能建议你每天整理一个 UI 组件库片段
- 你是产品经理？大模型可能建议你每天写一条用户洞察笔记
- 你是学生？大模型可能建议你每天把一个知识点用自己的话讲一遍

**不是通用的建议，而是为你量身定制的路径。**

#### 去问你的大模型

别直接抄我的答案。

去问你熟悉的大模型，问上面那个问题，看看它给你什么。

这就是千人千面的真正价值：

- **同一个 Prompt，不同的人，得到不同的答案**
- **每个答案都是基于你个人的背景、偏好、目标生成的**
- **它比你更了解你自己**

大模型不是在给你分类，而是在读你、理解你、然后给出最适合你的建议。
