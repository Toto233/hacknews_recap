## 3. 内容结构（演讲主线）

### 3.1 开场：什么是 vibe coding？

#### 核心理念
- 早上手，早受益
- 工具的复利效应
- 做自己真正用的东西，吃自己的狗粮
- 真正提高生产力，学会跟机器配合

#### 在 vibe coding 之前，AI 也已经能让人变强

即使还在 **前 Agent 时代**，只是认真地用 ChatGPT / Claude 提问，
也已经足够带来很大的成长。

同一个 Prompt，不同的人，会得到完全不同的答案。

> **"2026 年，如果只选一个习惯——只要做到每天稳定执行，就会最大程度上改变你的人生——这个习惯是什么？"**

你会得到一个更贴近你背景的建议，然后真正重要的是：
- 把它拆成「每天都能做完」的小动作
- 把结果固化成**可复用的东西**（代码 / 笔记 / 脚本 / 流程 / 文档）

所以 AI 最早改变我们的，未必是“替我们做事”，
而是先改变我们的**学习方式、思考方式和成长速度**。

#### 不是什么
- ❌ 完美主义
- ❌ 炫技式项目
- ❌ 为了学而学

---

### 3.2 工具生态：按「过去 / 现在 / 未来」理解

> 同一件事（写代码 / 做事情），不同阶段的工具关注点不同。

#### 过去：Cursor / Trae（AI IDE / 端到端工程师）

**Cursor - AI 原生 IDE**
- 基于 VS Code 二次开发
- 支持多模型：OpenAI、Anthropic、Gemini、xAI、Cursor
- 完整代码库理解能力
- 自主性滑块：从 Tab 补全到完全自主 agent

**三种使用模式：**
1. Tab 补全 - 像写代码一样自然
2. Cmd+K - 针对性编辑
3. Agent 模式 - 完全自主（构建、测试、演示）

**实际效果：**
- NVIDIA：40,000 工程师全面使用
- Salesforce：90% 开发者使用，PR 速度/质量双位数提升
- Y Combinator：最佳工程师都在用

**Trae - 你的 10x AI 工程师**
- Understand. Execute. Deliver.（理解 → 执行 → 交付）
- 强调端到端交付：理解需求 → 独立编码 → 完整交付

---

#### 现在：Claude Code（终端里的编程搭档）

**为什么是 Claude Code？**
- 代码能力强、上下文理解好
- 不是补全工具，而是工程搭档（读仓库 / 跑命令 / 产出 diff / 验证闭环）

**实战演示方向：**
- 环境部署
- 常见问题
- 实际案例

---

#### vibe coding 的最佳实践（把 AI 变成“可复利的系统”）

- 让 AI 帮你快速实现想法
- 不要追求完美，先跑起来
- 在迭代中学习 AI 的思维方式

**关键补丁：用 Git 把“每一次成功”存下来**

我们遇到过这种典型事故：
- 大模型因为**编码/换行/文件格式**等小问题，把项目一通改完
- 看起来改动很大、很“聪明”，但最后**完全运行不起来**

解决思路不是“别让 AI 改”，而是把风险变成流程：
- 每完成一个可运行状态就 **commit 一次**（保存 checkpoint）
- 大改动走 **分支**（feature/*），随时可回滚
- 先看 diff / 再跑测试 / 再合并（把“可运行”作为硬门槛）

> 原则：把一次性对话，变成可重复的流程；把不可控的风险，变成可回滚的版本。

### 3.3 OpenClaw：从 IDE 到个人助理（压轴）

> Claude Code 帮你写代码，OpenClaw 帮你做一切。

#### 3.3.1 什么是 OpenClaw？

**定位：**
- 个人 AI 助手框架
- 不止写代码，还能操作你的“数字生活”
- 24/7 待命，随时响应

**核心能力：**
- **消息与沟通：** 多渠道接入（Telegram/Discord/飞书/WhatsApp/Slack/Signal…），主动检查与提醒
- **文件与知识：** 本地文件读写、飞书文档、长期记忆管理
- **自动化能力：** Shell、浏览器自动化、节点控制（手机/摄像头/屏幕…）、定时任务（heartbeat / cron）

---

#### 3.3.2 模型切换：GLM-4.7 → ChatGPT 5.2（性格大变）

同一套使用方式，换模型之后“助理的性格”会明显变化：

- **GLM-4.7：** 更像一个美国人（更直接、更快、更偏执行）
- **ChatGPT 5.2：** 更像一个德国人（更严谨、更爱确认、更重视边界）

**结论：** 不是你变了，是模型的“气质/偏好”变了。

---

#### 3.3.3 真实案例：从「搞坏 WSL」到安卓手机上的 Dr.Two

以及另一个更“反直觉”的案例：把学习也做成 vibe coding。

- **Macroeconomics-for-Dummies-study（开源）**：用 **Codex / Claude Code** 把“学习一本书”工程化
  - 过程 repo 化：`progress/`、`summaries/`、`sessions/`
  - 仓库里有一个 `AGENTS.md`，可以直接复刻（学习助教的角色、提问方式、输出标准）
  - 每次学习都沉淀可复用产物：章节摘要 / 自测题 / 复习块
- 仓库： https://github.com/Toto233/Macroeconomics-for-Dummies-study
- 灵感来源（学习资料补充）：https://github.com/chenran818/CFP-Study/blob/main/CLAUDE.md

（配图：把“学习也做成工程”的一个小例子——GDP 统计口径里，卖不掉的产出会作为“存货投资”计入 GDP。）

![GDP/Inventory 例子：卖不掉也会计入 GDP](/presentation/images/macro-gdp-inventory.png)

我从今年 2 月初开始用 OpenClaw，到现在刚好一个月。

- **第一世：WSL Ubuntu**
  - 配置很快，但也很容易“用坏”
  - 让它做任务时把配置文件搞乱 → 最后整个 WSL 直接起不来

- **第二世：安卓手机 Termux（Dr.Two）**
  - 吃过教训后形成机制：
    1) 配置修改前先备份
    2) 重要修改先在测试环境试
    3) 建立守护进程监控 gateway 状态

---

#### 3.3.4 最近一次折腾：更新版本 + 换模型 → 行为大变

这段其实就是 3.3.2 的“现实版”：

- 因为 **OpenClaw 更新版本** + **更换模型（GLM-4.7 → ChatGPT 5.2）**，助理的行为习惯突然变了：
  - 以前更“直接执行”
  - 现在更“严谨确认”（每一步都想问一句）

我们真正做的不是“吐槽模型”，而是把它重新调回可用：

- 找到导致频繁确认/阻塞的配置开关
- 把不需要的确认开关关掉，把需要的边界留住
- 最终让它恢复到“该干活就干活”的状态（昨晚才调到正常）

**这一段的 vibe coding 含义：**
> 工具会变、模型会变，但你的系统要通过配置和文档把“稳定性”固化下来。

---

#### 3.3.5 上限极高，下限羸弱（以及为什么它仍然是未来）

- **上限极高：** 一旦开始用，会不断发现新可能性（发邮件 → 监控服务 → 管理工作流）
- **下限羸弱：** 配置文件错了就崩，一句话没说清就做错事

关键是：**这是一个会成长的东西**——用得越久、规则越清楚、文档越完善，它就越像你的专属助理。

## 4. 行动号召

也许我们还可以再等等。
等模型更强一点，等产品更成熟一点，等门槛再低一点。

这些事大概率都会发生。
但如果只是等，我们错过的，可能不只是更早上手的时间，
也是一次亲身经历：
看着 AI 从一个还不太可靠的助手，慢慢变成真正能和你一起创造东西的伙伴。

前面我讲了很多个人视角下的变化：
- 学习可以更快
- 写代码可以更快
- 知识积累可以更快
- 自动化能力也可以越来越强

所以如果只看个人体验，一个结论其实很容易成立：
**认真学习 AI、持续使用 AI，生产力大概率会显著提高。**

因为你不是只多了一个工具，
而是在慢慢建立自己的复利系统：
- 代码
- 笔记
- 脚本
- 流程
- 文档

但如果把视角再拉高一点，到公司、到组织、到整个经济层面，问题就没这么简单了。

这时候我想反问一句：

## 人工智能能提高生产力

**吗？**

这里就要提到一个很有意思的视角，叫 **索洛悖论（Solow paradox）**。
它最经典的一句话是：

> **你能在任何地方看到计算机时代，除了生产率统计里。**

翻成今天的话就是：
**技术看起来很强，但收益不一定立刻体现在结果上。**

后来经济学家和管理学者补上的关键解释是：

> **新技术不是装上就自动变成生产率，组织、流程、激励和管理方式也得一起改。**

我更喜欢把它说成一句人话：
- **技术像新引擎**
- **管理方式像旧车架**

你只换发动机，不换方向盘、刹车、传动和驾驶习惯，车不一定更快，甚至可能更难开。
AI 也是一样。
模型能力再强，如果公司还是老流程、老 KPI、老审批、老部门墙，很多潜力都会被锁死。

所以索洛悖论对今天 AI 的启发，不是“AI 没用”，而是：
1. **别把模型能力直接等同于经济产出**
2. **组织改造，往往比模型本身更稀缺**
3. **旧的管理方式，真的会卡住新的生产力**

所以最后我想把这件事收成两句话：

- **对个人来说：把 AI 变成你的复利系统。**
- **对企业来说：AI 不是给旧流程加涡轮，而是逼你重画流程图。**

我们都猜过，未来的软件开发大概会变成什么样。
但对今天坐在这里的人来说，
更重要的也许不是猜得多准，
而是有没有亲手参与，把那个未来一点点做出来。

**预测未来的最好方法，就是去创造它。**

**三步走：**
1. **了解工具**（Cursor / Trae / Claude Code / OpenClaw）
2. **快速上手**（今天就用起来）
3. **深入使用**（把工作流写进规则和文档里）
