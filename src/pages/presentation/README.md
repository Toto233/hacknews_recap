### 3.2 工具生态：按「过去 / 现在 / 未来」理解

> 同一件事（写代码 / 做事情），不同阶段的工具关注点不同。

#### 过去：Cursor / Trae（AI IDE / 端到端工程师）

**Cursor - AI 原生 IDE**
- 基于 VS Code 二次开发
- 支持多模型：OpenAI、Anthropic、Gemini、xAI、Cursor
- 完整代码库理解能力
- 自主性滑块：从 Tab 补全到完全自主 agent

**三种使用模式：**
1. Tab 补全 - 像写代码一样自然
2. Cmd+K - 针对性编辑
3. Agent 模式 - 完全自主（构建、测试、演示）

**实际效果：**
- NVIDIA：40,000 工程师全面使用
- Salesforce：90% 开发者使用，PR 速度/质量双位数提升
- Y Combinator：最佳工程师都在用

**Trae - 你的 10x AI 工程师**
- Understand. Execute. Deliver.（理解 → 执行 → 交付）
- 强调端到端交付：理解需求 → 独立编码 → 完整交付

---

#### 现在：Claude Code（终端里的编程搭档）

**为什么是 Claude Code？**
- 代码能力强、上下文理解好
- 不是补全工具，而是工程搭档（读仓库 / 跑命令 / 产出 diff / 验证闭环）

**实战演示方向：**
- 环境部署
- 常见问题
- 实际案例

---

#### vibe coding 的最佳实践（把 AI 变成“可复利的系统”）

- 让 AI 帮你快速实现想法
- 不要追求完美，先跑起来
- 在迭代中学习 AI 的思维方式

> 原则：把一次性对话，变成可重复的流程。

### 3.3 OpenClaw：从 IDE 到个人助理（压轴）

> Claude Code 帮你写代码，OpenClaw 帮你做一切。

#### 3.3.1 什么是 OpenClaw？

**定位：**
- 个人 AI 助手框架
- 不止写代码，还能操作你的“数字生活”
- 24/7 待命，随时响应

**核心能力：**
- **消息与沟通：** 多渠道接入（Telegram/Discord/飞书/WhatsApp/Slack/Signal…），主动检查与提醒
- **文件与知识：** 本地文件读写、飞书文档、长期记忆管理
- **自动化能力：** Shell、浏览器自动化、节点控制（手机/摄像头/屏幕…）、定时任务（heartbeat / cron）

---

#### 3.3.2 模型切换：GLM-4.7 → ChatGPT 5.2（性格大变）

同一套使用方式，换模型之后“助理的性格”会明显变化：

- **GLM-4.7：** 更像一个美国人（更直接、更快、更偏执行）
- **ChatGPT 5.2：** 更像一个德国人（更严谨、更爱确认、更重视边界）

**结论：** 不是你变了，是模型的“气质/偏好”变了。

---

#### 3.3.3 真实案例：从「搞坏 WSL」到安卓手机上的 Dr.Two

我从今年 2 月初开始用 OpenClaw，到现在刚好一个月。

- **第一世：WSL Ubuntu**
  - 配置很快，但也很容易“用坏”
  - 让它做任务时把配置文件搞乱 → 最后整个 WSL 直接起不来

- **第二世：安卓手机 Termux（Dr.Two）**
  - 吃过教训后形成机制：
    1) 配置修改前先备份
    2) 重要修改先在测试环境试
    3) 建立守护进程监控 gateway 状态

---

#### 3.3.4 这几天我们具体折腾了什么（可讲的部分）

这些都属于 vibe coding 的“日常”：不是写出一个完美系统，而是不断把坑填平、把规则固化。

- **演讲站点的 slides 混乱：** 同一个内容有好几套入口，改了 MD 但 slide 还是旧的
  - 解决：只保留 `presentation/` 目录下 **2 个 MD**（讲稿 + slides 源），其余全部删掉，避免分叉
- **线上页面 404/缓存：** 看起来“老”，其实是路由没对应到静态文件，或 CDN 没刷新
  - 解决：把演讲用的 Reveal 页面和 slides 源明确发布为静态资源
- **QQ 邮箱能力波动：** 改密码后授权失效，IMAP 登录失败
  - 解决：重新生成授权码后再更新配置
- **Win10 离线装 Python 包：** 只下主包会触发 pip 联网拉依赖；聊天工具下载会把 wheel 文件名改坏导致 pip 报错
  - 解决：一次性打包依赖 wheels；并把 wheel 文件名改回规范格式（否则会提示 not a supported wheel）

---

#### 3.3.5 上限极高，下限羸弱（以及为什么它仍然是未来）

- **上限极高：** 一旦开始用，会不断发现新可能性（发邮件 → 监控服务 → 管理工作流）
- **下限羸弱：** 配置文件错了就崩，一句话没说清就做错事

关键是：**这是一个会成长的东西**——用得越久、规则越清楚、文档越完善，它就越像你的专属助理。

---

### 3.4 真正的千人千面：让大模型给你定制建议

#### 一个 Prompt，无限可能

> **"2026 年，如果只选一个习惯——只要做到每天稳定执行，就会最大程度上改变你的人生——这个习惯是什么？"**

得到的建议可以是：每天产出一个「可积累的确定性成果」。

可以是：
- 一段 100 行以内的代码
- 一篇 300–500 字的笔记
- 一个流程优化脚本
- 一条可复用的 Prompt
- 一页讲清楚一件事的文档

关键不是建议本身，而是：**它会根据你的背景生成“你的路径”**。

## 4. 附录：Claude Code 提效要点（CLAUDE.MD）

> 想把 Claude Code 用到“像团队成员”，就要把规则写进 CLAUDE.MD。

### 4.1 CLAUDE.MD 的关键原则

1) **明确技术栈和约定**
```
❌ 错误：使用测试框架
✅ 正确：使用 pytest，测试文件命名规范为 test_*.py
```

2) **指定工具和命令**
```
❌ 错误：运行测试
✅ 正确：运行 pytest -v --cov=src
```

3) **代码风格约定**
```
❌ 错误：遵循 PEP 8
✅ 正确：使用 black 格式化，line length = 88，import 排序使用 isort
```

4) **项目结构说明**
```
项目结构：
- src/ 源代码
- tests/ 测试代码
- docs/ 文档
- scripts/ 脚本工具
```

### 4.2 版本控制安全原则

- ⚠️ 禁止自动 commit/push
- ⚠️ 先显示 diff，等待确认
- ⚠️ 修改前先创建分支

（下面保留一个可直接复制的示例）

```markdown
# 项目说明

## 技术栈
- Python 3.11
- FastAPI 0.104+
- PostgreSQL 15

## 开发约定

### 测试
- 使用 pytest
- 测试覆盖率目标：80%+
- 测试命令：pytest -v --cov=src

### 代码风格
- Black formatter (line length = 88)
- isort for imports
- 类型检查：mypy src/

### 版本控制
- ⚠️ 禁止自动 commit/push
- ⚠️ 任何 git 操作前必须获得明确授权
- ⚠️ 修改前先创建分支
- ⚠️ commit 前显示完整 diff 等待确认

### 项目结构
- src/ 核心代码
- tests/ 测试代码
- scripts/ 工具脚本
- docs/ 文档
```
